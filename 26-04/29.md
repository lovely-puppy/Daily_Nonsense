  **synchronized 怎么保证可见性？**
  > synchronized 通过内存语义保证可见性：线程加锁时会清空工作内存从主内存重新读取共享变量，解锁时会把工作内存中的修改刷回主内存。这样下一个获得锁的线程一定能读到上一个线程的最新修改。跟 volatile 的区别是 synchronized 还保证原子性，volatile 不保证原子性，所以 count++ 这种操作必须用 synchronized 或 Atomic 类，volatile 保护不了。

  **synchronized锁升级了解吗？**
  > synchronized的锁会根据竞争情况从低到高自动升级：无锁 —> 偏向锁 —> 轻量级锁 —> 重量级锁。只能升级不能降级，竞争越激烈锁越重。

  > 早期 JDK 中 synchronized 直接用重量级锁（操作系统的互斥量），每次加锁解锁都要从用户态切换到内核态，性能很差。但实际上很多场景下根本没有竞争或者竞争很轻微，没必要直接上重量级锁。
  
  > 无锁：没有线程来加锁
  > 偏向锁：只有一个线程用。第一个线程来加锁时，不真正加锁，只是在对象头里记一下这个线程的 ID，表示"这把锁偏向你了"。后续同一个线程再次加锁时，发现偏向的就是自己，直接进入，不需要任何同步操作。
  > 轻量级锁：两个线程交替用，没有同时竞争。
  > 重量级锁：竞争激烈的情况。自旋拿不到锁（竞争激烈），升级为重量级锁。线程进入操作系统的阻塞队列，让出 CPU，等锁释放了被唤醒。涉及用户态和内核态的切换，开销最大。
  
  **Synchronized和Reentrantlock有什么区别？**
  > ynchronized 是 JVM 关键字，自动加锁释放锁，使用简单但功能有限。ReentrantLock 是 Java API，需要手动 lock/unlock，但提供了更多能力：支持公平锁、可以 tryLock 尝试加锁、可以设超时时间、等待期间可以被中断、支持多个 Condition 实现精准唤醒。synchronized 有锁升级优化（偏向锁→轻量级锁→重量级锁），性能上两者差别不大。简单场景用 synchronized，需要高级功能用 ReentrantLock。两者都是可重入锁。
  
  **AQS了解吗？**
  > AQS 是一个抽象类，它维护了一个共享变量 state 和一个线程等待队列，为 ReentrantLock 等类提供底层支持。
  > AQS 的思想是，如果被请求的共享资源处于空闲状态，则当前线程成功获取锁；否则，将当前线程加入到等待队列中，当其他线程释放锁时，从等待队列中挑选一个线程，把锁分配给它。
  
  **说说ReentrantLock的实现原理**
  > ReentrantLock 底层基于 AQS 实现。AQS 内部有一个 state 变量和一个 CLH 等待队列。加锁时用 CAS 尝试把 state 从 0 改成 1，成功就持有锁，失败就进入等待队列阻塞。释放锁时 state 减 1，减到 0 就唤醒队列中下一个线程。可重入的实现是同一个线程重复加锁时判断持有者是自己就 state 累加，释放时递减到 0 才真正释放。公平锁和非公平锁的区别在于加锁时是否先检查等待队列，非公平锁先 CAS 抢一次抢不到再排队，公平锁直接排队。
  
  **CAS了解多少？**
  > CAS 是一种乐观锁，用于比较一个变量的当前值是否等于预期值，如果相等，则更新值，否则重试。
  
  - CAS的问题：ABA问题、自旋开销大、一次只能操作一个变量。
  
  **线程死锁**
  > 死锁发生在多个线程相互等待对方释放锁时。
  > 死锁是两个或多个线程互相持有对方需要的锁导致全部阻塞。排查方式首选 jstack 命令，打印线程堆栈后它会直接输出 "Found deadlock" 并显示哪些线程死锁、各自持有什么锁、在等什么锁、代码位置。也可以用 jconsole 可视化工具检测，或者在代码中用 ThreadMXBean 编程检测。预防死锁的方式主要有：固定加锁顺序避免循环等待、用 tryLock 设超时时间避免死等、缩小锁的范围减少持锁时间、尽量避免嵌套加锁。
  
  **说一说ConcurrentHashMap的实现原理**
  > ConcurrentHashMap 在 JDK 7 用分段锁实现，把 Map 分成 16 个 Segment，每个 Segment 独立加锁，不同段的操作可以并发。JDK 8 完全重写了，数据结构跟 HashMap 一样是数组+链表+红黑树，并发控制改成了 CAS + synchronized 锁单个桶的头节点。put 时如果桶是空的用 CAS 无锁写入，桶不是空的就 synchronized 锁住这一个桶的头节点再操作，锁粒度从一段变成了一个桶，并发度大幅提升。get 操作完全不加锁，因为 Node 的 value 和 next 用 volatile 修饰保证了可见性。相比 Hashtable 锁整张表的方式，性能好得多。
  
  - CopyOnWriteArrayList的实现原理。
  > CopyOnWriteArrayList时ArrayList的线程安全版本，适用于读多写少的场景。
  > 核心思想是写的时候创建一个新数组，修改后替换掉原数组，这样就能确保读操作无锁（读旧数据），从而提高并发性能。内部使用volatile变量修饰数组array，保证读操作的内存可见性。
  
  - 说一下对BlockingQueue的理解
  > BlockingQueue是阻塞队列，当队列满了会阻塞等待，队列空了也会阻塞等待，天然适合生产者-消费者模型。
  > 常见实现有 ArrayBlockingQueue（有界数组，一把锁）、LinkedBlockingQueue（有界/无界链表，两把锁，吞吐量更高）、SynchronousQueue（不存储元素，直接传递）、DelayQueue（延迟队列）。线程池的工作队列就是 BlockingQueue，核心线程忙不过来时任务就放进队列排队。
  
  **什么是线程池？**
  > 线程池就是提前创建好一批线程放在池子里进行复用，有任务来了就取一个线程去执行，执行完了不销毁放回池子里等待下一个任务。避免了频繁创建和销毁线程的开销。
  
  - 线程池的七个核心参数：
  - 核心线程数
  > 始终存活不会被回收
  
  - 最大线程数
  > 核心加非核心的上限
  
  - keepAliveTime
  > 非核心线程空闲多久后会被回收
  
  - unit
  > 存活时间的单位
  
  - workQueue
  > 工作队列，核心线程忙时任务在这里排队
  
  - threadFactory
  > 创建线程的工厂，可以自定义线程名
  
  - handler
  > 拒绝策略，线程和队列都满了怎么办
  
  - 线程池的工作流程
  > 先任务提交，然后检查核心线程是否有空闲，有就直接执行，没有就减产工作队列满没有，没满就放进工作队列等待，满了就检查是否达到最大线程数，没有就创建非核心线程执行，满了就执行拒绝策略。
  
  - 四种拒绝策略
  1. AbortPolicy(默认)，直接抛异常，适合不能丢任务的场景
  2. CallerRunsPolicy，让提交任务的线程自己执行，适合不想丢任务又不想抛异常
  3. DiscardPolicy，直接丢弃不抛异常不做处理，适合可以丢任务的场景
  4. DiscardOldestPolicy，丢弃队列里最老的任务然后尝试重新提交被拒绝的任务，适合新任务比老任务重要的场景
  
  - 核心线程怎么设置
  > 对于IO密集型任务，核心线程数设置为CPU核数×2，线程经常等IO任务，多开几个线程不让CPU闲下来。
  > 对于CPU密集型的任务，核心线程数设置为CPU核数+1，线程一直在占用CPU，线程多了也拿不到CPU，反而增加切换开销。
  
  - 四种常见的线程池：
  1. FixedThreadPool(固定大小线程池)，固定5个线程（核心线程数等于最大线程数），多余任务去队列排队，队列是LinkedBlockingQueue无界队列，任务堆积会OOM
  2. CacheThreadPool（缓存线程池），核心线程数为0，最大线程数无限（Integer.MAX_VALUE），来一个任务创建一个任务，空闲60秒自动回收
  3. SingleThreadExecutor(单线程池)，只有一个线程，所有任务排队串行执行，保证任务按提交顺序执行。 队列也是无界的，同样可能 OOM。
  4.  ScheduledThreadPool（定时线程池）
  **但是这四种线程池都是用Executors创建线程池，都有OOM的风险，不推荐使用，要用ThreadExecutor手动创建**
 
  - 线程池的几种阻塞队列
  > 常用的有五种，有界队列 ArrayBlockingQueue；无界队列 LinkedBlockingQueue；优先级队列 PriorityBlockingQueue；延迟队列 DelayQueue；同步队列 SynchronousQueue。
  
  **线程池调优了解吗？**
  > 线程池调优主要调的是核心线程数、最大线程数、工作队列大小，根据任务类型和业务场景来配。
