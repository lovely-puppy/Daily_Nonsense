  **进程和线程的区别？**
  > 进程是资源分配的最小单位，线程是 CPU 调度的最小单位。 一个进程可以包含多个线程，线程共享进程的堆内存和方法区，但每个线程有自己私有的栈和程序计数器。进程之间内存隔离互不影响，一个崩了不影响其他的；线程之间共享内存，一个线程崩了整个进程都崩，而且共享数据需要加锁保证线程安全。进程创建和切换开销大，线程相对轻量。进程间通信要靠管道、消息队列、共享内存、Socket 等机制，线程间直接读写共享变量就行。使用多线程的主要目的是提高 CPU 利用率和响应速度。
  
  **线程有几种创建方式？**
  > 继承Thread类：简单直接，但 Java 是单继承，继承了 Thread 就不能继承其他类了。不推荐。
  > 实现Runnable接口：实现接口不影响继承其他类，而且多个线程可以共享同一个 Runnable 对象。推荐。
  > 实现 Callable + FutureTask（有返回值）：跟 Runnable 最大的区别：Callable 有返回值，Runnable 没有。而且 Callable 的 call() 方法可以抛异常，Runnable 的 run() 不行。
  > 线程池（实际开发中用的方式）：实际开发中几乎只用线程池，不会手动 new Thread。因为频繁创建销毁线程开销大，线程池可以复用线程、控制并发数量、统一管理。
  
  **你平时有用过多线程吗？你在代码中是哪些场景用呢？**
  > 在项目中我主要在三个场景用到了多线程。第一个是异步处理，用户下单后核心操作同步执行，发短信、送积分这些非核心操作丢到线程池异步执行，减少用户等待时间。第二个是并行查询，首页需要调多个服务获取数据，用 CompletableFuture 并行调用，总耗时从串行的累加变成取决于最慢的那个。第三个是批量数据处理，大量数据拆成多批用线程池并行插入数据库，提升处理速度。线程池用的是 Spring 的 ThreadPoolTaskExecutor，核心线程数根据 CPU 核心数和任务类型来配置，IO 密集型设大一些，CPU 密集型设成 CPU 核数 + 1。\
  > CPU 密集型任务（计算、排序）：核心线程数 = CPU 核数 + 1
  > IO 密集型任务（数据库查询、网络请求）：核心线程数 = CPU 核数 × 2
  
  **调用 start 方法时会执行 run 方法，那怎么不直接调用 run方法？**
  > 直接调用 run() 方法只是一个普通的同步方法调用，所有代码都在当前线程中执行，不会创建新线程。没有新的线程创建，也就达不到多线程并发的目的。
  
  **线程有哪些常用的调度方法？**
  > 线程常用的调度方法有：sleep 让当前线程暂停指定时间但不释放锁；yield 让出 CPU 但只是建议不保证生效；join 等待另一个线程执行完再继续；wait 释放锁并等待通知，必须在 synchronized 里调用；notify/notifyAll 唤醒等待的线程。其中 sleep 和 wait 是最常考的对比：sleep 是 Thread 的静态方法不释放锁，wait 是 Object 的方法释放锁，sleep 到时间自动醒，wait 靠 notify 唤醒。
  
  - wait 为什么必须在 synchronized 里?
  > 因为 wait/notify 是线程间通信的机制，必须保证"检查条件"和"进入等待"是原子操作。如果不在 synchronized 里，可能出现：线程 A 检查完条件发现要 wait，还没来得及 wait，线程 B 就已经 notify 了，A 再 wait 就永远等不到通知了（信号丢失）。
  
  **线程有几种状态？**
  > 6种
  
  - NEW: 线程对象创建了，但还没调start()
  - RUNNABLE（运行态）：调了 start() 之后进入这个状态。线程可能正在 CPU 上跑，也可能在就绪队列里等 CPU 分配时间片。
  - BLOCKED（阻塞态）：线程想进入 synchronized 块，但锁被别人占了，只能等。
  - WAITING（等待态）：线程无限期等待，必须被别人主动唤醒才能恢复。
  - TIMED_WAITING（超时等待）：跟 WAITING 类似，但有时间限制，到时间自动醒。
  - TERMINATED（终止）：线程的 run() 方法执行完了，或者抛了未捕获的异常，线程死亡。死了就不能再 start() 了。

  **请说说sleep和wait的区别？**
  > sleep 是 Thread 的静态方法，让当前线程暂停指定时间，不释放锁，时间到了自动醒。wait 是 Object 的实例方法，必须在 synchronized 块里调用，调用后释放锁进入等待状态，需要被 notify 或 notifyAll 唤醒。最本质的区别是 sleep 不释放锁，wait 释放锁。sleep 用于单纯的暂停，wait 用于线程间的等待-通知协作。
  
  **怎么保证线程安全？**
  - 使用synchronized关键字对方法或代码块进行加锁，保证同一时间只有一个线程能够执行。
  - Reentrantlock：比 synchronized 多了几个能力：可以尝试加锁（tryLock）、可以设超时时间、可以中断等待、支持公平锁和非公平锁。
  - 使用volatile保证可见性和禁止指令重排序，但不保证原子性。
  - Atomic原子类：底层用 CAS（Compare And Swap） 实现，不加锁，性能比 synchronized 高。适合简单的计数、累加场景。
  - 使用线程安全的集合：ConcurrentHashMap 用分段锁 / CAS 实现高并发读写。CopyOnWriteArrayList 写的时候复制一份数组，读写不冲突，适合读多写少场景。
  - 使用ThreadLocal：不是加锁，而是让每个线程持有自己独立的变量副本，各用各的互不干扰。典型场景是 SimpleDateFormat（线程不安全的类）、数据库连接、用户登录信息。
  
  **公平锁和非公平锁**
  > 公平锁按申请顺序排队，先来先得，不会饿死但性能差，因为每次释放锁都要唤醒队列中的线程，唤醒有上下文切换开销。非公平锁允许插队，新来的线程先尝试直接获取锁，抢到就执行，抢不到再排队。非公平锁吞吐量更高，因为减少了线程唤醒的等待时间。ReentrantLock 默认是非公平锁，传 true 变成公平锁，synchronized 只有非公平锁。大部分场景用非公平锁就够了，只有对公平性要求很高的场景才用公平锁。
  
  **什么是ThreadLocal？**
  > ThreadLocal 是线程的私有变量，每个线程都有自己的独立的一份副本，线程之间互不影响。
  
  > 每个Thread对象内部有一个ThreadLocalMap对象，是一个以ThreadLocal为key，以设的值为value的Map。
  
  > ThreadLocal 是线程级别的私有变量，每个线程持有自己独立的副本，线程之间互不影响。底层实现是每个 Thread 对象内部有一个ThreadLocalMap，调 set 时往当前线程的 Map 里存，调 get 时从当前线程的 Map 里取。常用场景有存储用户登录信息（请求进来存，整个调用链都能取）、解决 SimpleDateFormat 线程不安全问题、以及 Spring 事务管理中保证同一线程用同一个数据库连接。需要注意内存泄漏问题，ThreadLocalMap 的 key 是弱引用，ThreadLocal 对象被 GC 后 key 变 null 但 value 还在，所以用完必须调 remove 清理，尤其是线程池场景。
  
  > 实际开发中 ThreadLocal 一般声明为 static 的，强引用一直存在，不会被回收，也就不会出现 key 为 null 的情况。内存泄漏主要出现在 ThreadLocal 声明为局部变量或者手动置 null 的场景。
  
  - 父线程能用ThreadLocal给子线程传值吗？
  > 不能，ThreadLocal是线程私有的，父线程存的值子线程拿不到。
  > 可以使用InheritableThreadLocal，创建子线程的时候把父线程的值复制一份给子线程。
  > TransmittableThreadLocal，解决InheritableThreadLocal不能解决线程池里线程复用的问题。InheritableThreadLocal 只在创建子线程时复制一次。线程池里线程是复用的，第二次任务用的是老线程，不会重新复制，拿到的还是第一次的旧值。 TransmittableThreadLocal（TTL），专门解决线程池场景下的值传递问题。它在提交任务时（而不是创建线程时）把值传递过去。
  
  **说一下你对Java内存模型的理解？**
  > 核心问题是多线程环境下，一个线程修改了变量，另一个线程可能看不到。JMM就定义了多线程下共享变量的读写规则。
  
  - JMM 解决的三个问题
  > 可见性：一个线程修改了变量，其他线程能不能立刻看到。
  > 解决方案：用 volatile 修饰。volatile 变量每次读都从主内存读，每次写都立刻刷回主内存。
  
  > 原子性：一个操作是不可分割的，要么全做要么全不做。
  > 解决方案：synchronized、Lock、Atomic原子类。
  
  > 有序性：代码按照我写的顺序执行。但 JVM 和 CPU 为了优化性能可能会指令重排序。
  > 解决方案：解决方案：volatile（禁止重排序）、synchronized（保证有序性）。
