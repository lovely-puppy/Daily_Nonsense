  **数据库的三大范式？**
  > 第一范式：列不可再分（字段原子性）
  > 第二范式：不能部分依赖，非主键列必须完全依赖于主键列
  > 第三范式：不能传递依赖，非主键列之间不能有依赖关系

  ** MySql有哪些常见的存储引擎？**
  - 默认是InnoDB存储引擎，还有MyISAM和Memory存储引擎
  > InnoDB支持事务、行锁、外键、聚簇索引、MVCC
  > MyISAM 不支持事务、行锁和MVCC、是非聚簇索引、支持表锁，查询速度快、适合读多写少的场景。
  > Memory 数据存储在内存中，重启就没了。速度极快，适合做临时表或缓存。

  **MySQL的日志文件有哪些？**
  > MySQL 有三个核心日志。redo log 是 InnoDB 引擎层的，记录数据页的物理修改，保证事务的持久性，宕机后通过 redo log 做崩溃恢复。undo log 也是引擎层的，记录修改前的旧值，保证原子性用于事务回滚，同时支持 MVCC 的历史版本读取。binlog 是 Server 层的，记录所有写操作，用于主从复制和数据恢复。事务提交时通过两阶段提交保证 redo log 和 binlog 的一致性：redo log 先 prepare，binlog 写完后 redo log 再 commit

  **为什么要两阶段提交？**
  > 两阶段提交是为了保证 redo log 和 binlog 的数据一致。如果两个日志不一致，主从复制或数据恢复时就会出现主库和从库数据不一样的问题。两阶段提交把 redo log 拆成 prepare 和 commit 两个阶段，先写 redo log prepare，再写 binlog，最后 redo log commit。宕机恢复时，如果 redo log 是 prepare 状态，就去查 binlog 有没有对应记录——有就提交，没有就回滚。这样不管在哪一步宕机，都能保证两个日志最终一致。

  - 为什么需要两个日志？一个日志不可以吗？
  > 只用 redo log 不行：redo log 是循环写的，会覆盖旧数据，不能用于主从复制和长期的数据恢复。binlog 是追加写的，所有历史操作都保留，主从复制必须靠它。

  > 只用 binlog 不行：binlog 没有崩溃恢复能力。它记录的是逻辑操作，不是数据页的物理修改，没办法恢复 Buffer Pool 中未刷盘的脏页。而且 binlog 是 Server 层的，不了解 InnoDB 引擎内部的数据页状态。

  **什么是sql慢查询？**
  > 慢 SQL 是执行时间超过阈值的 SQL，通过慢查询日志(slow query log)记录。常见原因有没走索引、索引失效、查询数据量太大、多表 JOIN 太多、深度分页等。排查时用 EXPLAIN 看执行计划，重点关注 type 是不是 ALL 全表扫描、key 有没有走索引、rows 扫描行数多不多。优化手段主要是加索引、避免 SELECT *、避免索引失效的写法、优化分页方式、用 JOIN 替代低效子查询。

  **如何优化慢SQL？**
  > 首先查看慢SQL日志定位到慢SQL语句

  > 然后使用explain进行分析

  > 对症优化：1、加索引；2、避免索引失效（左模糊，不遵守最左前缀，隐式类型转换，对索引列用函数）；3、避免select \*；4、优化深度分页；5、优化join；6、用join代替子查询；7、优化group by和order by；8、批量操作代替逐条操作
  
  **explain平常有用过吗？**
  > explain主要有四个字段，type是访问类型，key是实际使用哪个索引，rows是查看扫描行数，extra是额外信息（看有没有using index）
  > type从好到差排序：system > const > eq_ref > ref > range > index > ALL
  
  - const：通过主键或唯一索引查一条数据，最快。
  - eq_ref：JOIN 时被驱动表用主键或唯一索引关联，每次只匹配一行。
  - ref：用普通索引查，可能匹配多行。
  - range：索引范围扫描，比如 >、<、BETWEEN、IN。
  - index：扫描整棵索引树，比全表扫描好一点，但也不理想。
  - ALL：全表扫描，最差。看到这个就要优化。
  
  > Extra: using index （覆盖索引）好
  > using where (在server层过滤) 正常
  > using index condition（索引下推）好
  > using filesort （额外排序，没有使用索引排序）需要优化
  > using temporary（用了临时表）需要优化
  
  - *优化案例*
  > 之前项目里有个查询用户列表的接口响应很慢，用 EXPLAIN 分析发现 type 是 ALL，key 是 NULL，rows 扫了 50 万行，说明完全没走索引。检查了一下发现 WHERE 条件里的 name 字段没有建索引，加了索引之后再看 EXPLAIN，type 变成了 ref，rows 降到了几行，响应时间从 3 秒降到了几十毫秒。
