  **cookie-session是什么和JWT有什么区别**
  - Cookie-Session 是服务端有状态方案：用户数据是存在服务端的 Session 里，客户端通过 Cookie 传递 Session ID。优点是服务端可控（随时注销），缺点是分布式场景需要共享Session。
  - Token（JWT）是无状态方案：用户信息直接编码在 Token 里由客户端持有，服务端只做签名验证。优点是天然支持分布式和跨端，缺点是无法主动注销，Token体积较大。

**说一下Redis和HaspMap的区别**
- Redis 和 HashMap都是键值对存储，HashMap是进程内的数据结构，Redis是独立的网络服务
- HashMap只能在当前服务器的JVM进程中使用。Redis是多台服务器共享的，多台服务器连接同一个redis，读写的是同一份数据。
- HashMap就是一个Map， 只有put/get/remove这些基本操作。Redis 支持五种数据结构（String、Hash、List、Set、ZSet），还有过期时间、发布订阅、Lua 脚本、事务、分布式锁等功能。比如做排行榜用 ZSet，做消息队列用 List 或 Stream，这些 HashMap 做不了。
- HashMap 本身线程不安全，多线程下要用 ConcurrentHashMap。Redis 是单线程模型处理命令，天然不存在并发问题，所有命令按顺序执行。
- HashMap是直接操作内从，而Redis会走网络通信，多一份IO开销。
- HashMap受限与JVM堆内存，Redis是独立进程，可以分配几十上百G内存。

**Redis的持久化方式有哪些？**
RDB(快照)和AOF（追加日志），都是解决redis数据在内存中，进程挂了数据就没了，需要一种机制来把数据持久化到磁盘中。
- RDB是对某一时刻的内存数据做一份全量快照，生成一个压缩的二进制文件。
触发方式：save(手动触发，会阻塞)，bgsave（fork子进程生成快照，不阻塞）
优点：体积小速度快，适合做备份和灾难恢复。缺点：会丢失数据
- AOF是把每一条命令都追加到日志文件末尾，重启时按顺序重放这些命令来恢复数据
always：每条命令立即刷盘，数据最安全，性能最差
everysec：每秒刷盘一次，最多丢一秒数据，性能良好
no：交给操作系统决定什么时候刷盘，可能会丢十几秒数据，性能最好
AOF重写：AOF文件记录了所有历史命令，文件会越来越大，AOF重写就是用当前内存数据反向生成一份精简指令集。
AOF优点：数据更安全，everysec最多丢失一秒数据，命令是文本文件，可读性好，易恢复
缺点：性能较差，文件体积比RDB大，恢复速度也更慢。
- "RDB 和 AOF 同时开启，Redis 重启时用哪个恢复？"
> 用	AOF，数据更完整

- "fork 子进程的时候，主进程有新的写操作怎么办？"
> 靠写时复制（copy-on-write），共享一份内存页

- "AOF 重写期间有新的写命令怎么办？"
> 在旧AOF文件和AOF重写缓冲区都追加上，子进程完成后把缓冲区的命令追加到新AOF文件的末尾然后替换掉旧AOF
